# 数据标注规范 v1.0

> 项目：基于小模型强化学习的语义风险推理系统  
> 版本：v1.0-draft  
> 用途：指导人工构造、审核、清洗、扩充训练数据，确保 SFT、DPO/ORPO/GRPO、评测集使用统一标准。  
> 数据对象：中文弹幕、评论、短帖、视频标题/简介关联评论、直播聊天文本等。

---

## 0. 标注目标

本项目不是训练一个“敏感词分类器”，而是训练一个能够进行语义风险推理的小模型。

每条标注数据应帮助模型学习：

1. 如何识别非字面表达。
2. 如何拆解编码机制。
3. 如何结合上下文判断风险。
4. 如何避免过度敏感。
5. 如何给出简洁、可审计、不过度脑补的解释。

---

## 1. 单条样本基本结构

推荐使用 JSONL 格式，每行一条样本。

```json
{
  "id": "C1_000001",
  "split": "train",
  "source_type": "synthetic_or_real",
  "platform": "bilibili",
  "text": "示例文本",
  "context": {
    "title": "可选：视频标题或帖子标题",
    "description": "可选：简介",
    "time": "2026-06-01",
    "topic": "可选：话题"
  },
  "risk_level": "high",
  "encoding_primary": "C1_历史人物类比",
  "encoding_secondary": ["D3_借代"],
  "needs_context": true,
  "ambiguity": "medium",
  "evidence_strength": "strong",
  "hard_negative": false,
  "target_reference": "可选：目标对象，必要时可匿名化",
  "should_explain_target": false,
  "reasoning": {
    "literal_analysis": [],
    "encoding_analysis": [],
    "context_analysis": [],
    "supporting_evidence": [],
    "counter_evidence": [],
    "final_rationale": ""
  },
  "quality_status": "approved",
  "review_notes": ""
}
```

---

## 2. 必填字段

| 字段 | 是否必填 | 说明 |
|---|---:|---|
| id | 是 | 唯一编号 |
| text | 是 | 待判断文本 |
| risk_level | 是 | high/medium/low/none |
| encoding_primary | 是 | 主编码方式；无风险样本可填 none |
| needs_context | 是 | 单看 text 是否无法稳定判断；默认 false |
| ambiguity | 是 | low/medium/high |
| evidence_strength | 是 | weak/moderate/strong |
| hard_negative | 是 | 是否困难负样本 |
| reasoning.literal_analysis | 是 | 字面层分析 |
| reasoning.encoding_analysis | 是 | 编码机制分析 |
| reasoning.context_analysis | 是 | 语境分析 |
| reasoning.supporting_evidence | 是 | 支持风险的证据 |
| reasoning.counter_evidence | 是 | 支持低风险或无风险的证据 |
| reasoning.final_rationale | 是 | 最终理由 |

## 3. 风险等级标注标准

### 3.1 high

标为 high 必须满足：

- 目标指向明确，或
- 多个证据互相印证，或
- 文本本身已经形成强风险，或
- 真实上下文把模糊文本提升为强风险，或
- 存在明确规避检测意图。

标注时必须能回答：

```text
它在表面说什么？
它可能实际表达什么？
为什么这种解释比普通解释更合理？
证据来自文本本身，还是来自真实可用上下文？
```

如果无法回答，不得标 high。
high 不强制要求上下文。弹幕、短评论、暗号、直白违规表达可以仅凭 text 判 high；不得为了把样本做成 high 而编标题、话题或互动背景。

本项目主要面向弹幕、应用评论、音乐评论等短文本。若某个事件或背景在真实平台里不可公开讨论、不可被模型输入观测，就不能拿它来解释一条普通短句。比如“勇士的精神值得我们学习”这类裸文本，在没有真实可见上下文时应按普通表达或 hard negative 处理，而不是靠标注者知道的敏感事件升为风险。

但稳定黑话、出圈梗、高频结构化代指本身就是文本证据，不需要额外上下文。例如某些简中鉴证圈高频句式已经形成固定映射，即使作为弹幕或评论孤立出现，也应按 high 训练。

---

### 3.2 medium

适用于：

- 有明显编码痕迹，但目标不够明确。
- 有风险解释和普通解释，两者都合理。
- 文本有风险信号，但需要更多上下文才能确认目标或严重程度。
- 证据强但只有单一来源。

medium 是“有风险信号但强度不足”，不是流程里的“待人工复核”。

---

### 3.3 low

适用于：

- 存在轻微暗示。
- 语言风格像隐晦表达，但缺少目标。
- 文本有潜在歧义，但普通解释更可能。
- 只出现弱信号，如“懂的都懂”“不好说”，但无风险语境。

---

### 3.4 none

适用于：

- 正常生活表达。
- 正常历史、文学、娱乐讨论。
- 普通错别字、缩写、玩梗。
- 关键词撞车但语境无风险。
- 仅有模型可能误读的表面词。

none 样本必须认真写 counter_evidence，不能只写“无风险”。

---

## 4. 编码方式标注

主编码方式只选一个，即对风险判断贡献最大的机制。

次编码方式可以多个。

示例：

```json
{
  "encoding_primary": "C1_历史人物类比",
  "encoding_secondary": ["D3_借代", "E1_平台黑话"]
}
```

如果没有编码机制：

```json
{
  "encoding_primary": "none",
  "encoding_secondary": []
}
```

---

## 5. 上下文标注规范

### 5.1 needs_context

默认标 false。只有单看 text 无法做出稳定判断，且上下文是真实、必要、非硬编时，才标：

```json
"needs_context": true
```

常见情况：

- “又到了”
- “懂的都懂”
- “不能说”
- “熟悉的味道”
- “历史总是相似”
- 单个数字、表情、外号

不要为了让样本显得完整而硬凑上下文。没有真实上下文就留空，让模型按裸文本判断并在证据不足时降级。

### 5.2 上下文最小化原则

只保留判断所需上下文，避免塞入无关长文本。

推荐顺序：

1. 标题。
2. 发布时间。
3. 话题标签。
4. 平台来源。
5. 真实原始上下文中的必要片段。

---

## 6. 推理链写法

推理链必须“短、准、可审计”。

禁止写成大段文学化解释。

推荐格式：

```text
字面层：这句话表面是在……
编码层：可疑点是……，对应的编码机制是……
语境层：没有上下文就写“无明确上下文”；有真实上下文时再说明它如何改变判断。
反证：普通解释可能是……，但因为……所以权重较低/较高。
结论：综合判断为 high/medium/low/none。
```

---

## 7. 必须包含反证

每条样本都必须写 counter_evidence。

因为本项目最重要的问题不是“漏判”，而是“模型过度联想”。

### 反证示例

```text
支持风险：
- 出现疑似历史类比。
- 与视频标题中的现实议题有关。

反证：
- 文本也可能是正常历史人物讨论。
- 没有直接出现目标对象或明确时间节点。
```

### 为什么必须写反证

这能让模型学会：

- 不要看到关键词就判高风险。
- 不要把普通历史讨论误杀。
- 不要无依据还原具体对象。
- 学会在证据不足时降级。

---

## 8. Hard Negative 规范

Hard Negative 是本项目的关键数据。

定义：表面上容易被误判为风险，但实际应为 none 或 low 的样本。

### 8.1 常见类型

| 类型 | 例子 | 正确标签 |
|---|---|---|
| 关键词撞车 | “我今天买了很多白纸” | none |
| 正常历史讨论 | “司马懿在后期政治斗争中很能忍” | none |
| 普通反讽 | “这游戏优化真是太棒了，卡成PPT” | none/low |
| 普通数字 | “64号选手表现不错” | none |
| 普通缩写 | “这个up主太nb了” | none |
| 普通外号 | “小熊维尼这集很好看” | none |
| 无上下文暗示 | “懂的都懂” | low |
| 娱乐玩梗 | “曹操来了都得点赞” | none |

### 8.2 配比要求

训练集建议：

```text
high: 25%
medium: 25%
low: 20%
none: 30%
```

其中：

```text
none 中 hard_negative ≥ 50%
low 中 needs_context ≥ 30%
```

### 8.3 构造方法

每个高风险样本至少配一个相似负样本。

例如：

```text
高风险样本：
文本：又到了那个不能说的夏天
上下文：历史事件相关视频评论区
标签：high

配对负样本：
文本：又到了那个不能说的夏天
上下文：恋爱综艺评论区，讨论嘉宾分手
标签：low/none
```

---

## 9. 歧义度 ambiguity

### low

- 解释基本唯一。
- 普通解释和风险解释差距明显。
- 标注者容易达成一致。

### medium

- 有两种解释。
- 需要上下文才能稳定判断。
- 标注者可能需要讨论。

### high

- 证据不足。
- 多种解释都合理。
- 不适合作为训练主样本，可放入评测集或人工复核池。

---

## 10. 证据强度 evidence_strength

### strong

- 至少两类独立证据互相印证。
- 文本和上下文共同支持。
- 目标指向明确。

### moderate

- 有明确可疑点。
- 但缺少一个关键上下文。
- 或只有单一强证据。

### weak

- 只有风格、语气或弱提示。
- 无明确目标。
- 普通解释更可能。

---

## 11. target_reference 与 should_explain_target

某些训练场景不希望模型直接输出具体敏感对象或事件。

因此拆分两个字段：

```json
"target_reference": "内部标注，可匿名化",
"should_explain_target": false
```

### 规则

- target_reference 用于内部训练分析、评估，不一定进入模型输出。
- 如果 should_explain_target=false，SFT 输出中只写“特定历史事件/现实政治人物/敏感公共事件”，不写具体名称。
- 如果任务目标是内容审核辅助，建议默认 false。
- 如果任务目标是研究分析，可在受控数据集里设为 true。

---

## 12. SFT 输出格式

推荐训练模型输出如下：

```text
## 字面层分析
...

## 编码机制分析
...

## 语境层分析
...

## 证据与反证
支持风险：
- ...
反证：
- ...

## 最终判定
risk_level: medium
encoding_primary: C1_历史人物类比
confidence: 0.68
rationale: ...
```

要求：

1. 不输出过长 chain-of-thought。
2. 不写无法验证的社会背景。
3. 不强行还原目标。
4. 不因一个关键词直接判 high。
5. 必须给出反证或不确定性。

---

## 13. DPO/ORPO 偏好数据规范

偏好数据用于训练模型边界。

格式：

```json
{
  "prompt": "请判断以下文本风险……",
  "chosen": "更好的回答",
  "rejected": "更差的回答",
  "reason": "chosen 好在哪里，rejected 错在哪里"
}
```

### 13.1 chosen 应具备

- 风险等级正确。
- 解释不过度。
- 能指出上下文不足。
- 有反证。
- 格式稳定。

### 13.2 rejected 常见错误

- 过度敏感。
- 无依据判 high。
- 看到关键词就联想。
- 格式混乱。
- 编造不存在的语境。
- 忽略上下文。
- 不输出反证。
- 直接还原敏感对象但证据不足。

---

## 14. GRPO 奖励函数设计建议

GRPO 不应主要奖励格式关键词。

推荐权重：

```text
准确性奖励：50%-60%
边界校准奖励：20%-30%
解释一致性奖励：15%-20%
格式奖励：5%-10%
```

### 14.1 应奖励

- 标签与人工标注一致。
- high/medium/low/none 边界合理。
- 能说明证据不足。
- 能给出普通解释。
- 上下文存在时能正确利用上下文。
- 上下文不足时不强行判 high。
- 输出格式稳定。

### 14.2 应惩罚

- 格式堆砌但内容空洞。
- 无依据还原具体对象。
- 把普通历史讨论判高风险。
- 把普通生活词汇判高风险。
- 忽略反证。
- 编造上下文。
- 输出过长、重复、模板化。
- 把所有隐晦表达都判 high。

## 15. 质量审核流程

### 15.1 三阶段审核

```text
初标
↓
复核
↓
抽检
```

### 15.2 初标

标注者完成：

- 标签。
- 编码方式。
- 推理链。
- 反证。
- 上下文字段。

### 15.3 复核

复核者检查：

- 标签是否过重。
- 是否有足够证据。
- 是否遗漏 hard negative。
- 推理是否跳步。
- target_reference 是否应隐藏。

### 15.4 抽检

每 100 条至少抽检 20 条。

重点抽检：

- high 样本。
- needs_context 样本。
- hard_negative 样本。
- C1/C2/C4 类语义编码样本。

---

## 16. 标注一致性标准

### 16.1 一致性目标

初期目标：

```text
risk_level 一致率 ≥ 80%
encoding_primary 一致率 ≥ 75%
high/none 二分类一致率 ≥ 90%
```

成熟后目标：

```text
risk_level 一致率 ≥ 85%
encoding_primary 一致率 ≥ 80%
high/none 二分类一致率 ≥ 95%
```

### 16.2 分歧处理

如果标注者分歧：

- high vs medium：检查是否目标明确。
- medium vs low：检查是否上下文足够。
- low vs none：检查是否存在真实编码机制。
- 任意等级跨度超过 2：进入仲裁池。

---

## 17. 数据集划分

推荐：

```text
train: 80%
validation: 10%
test: 10%
```

但必须按“语义簇”划分，不能简单随机。

### 17.1 避免泄漏

同一梗、同一模板、同一目标对象、同一改写簇不能同时出现在 train 和 test。

错误做法：

```text
train: 又到了那个不能说的夏天
test: 又到那个不能说的月份了
```

这是模板泄漏。

### 17.2 推荐测试集类型

| 测试集 | 用途 |
|---|---|
| ID Test | 同分布能力 |
| OOD Test | 新对象、新语境 |
| Hard Negative Test | 误杀率 |
| Context Flip Test | 上下文反转能力 |
| Novel Encoding Test | 新编码机制泛化 |
| Real Sample Test | 真实平台可用性 |

## 19. 最小可行数据集 MVP

第一版不要直接做 3000 条。

建议先做 1000 条：

```text
high: 250
medium: 250
low: 200
none: 300
```

其中：

```text
hard_negative: ≥ 150
needs_context: ≥ 250
每个一级编码类别: ≥ 100
每个重点二级类别: ≥ 40
```

MVP 目标不是一次训出最佳模型，而是验证：

1. 4B/3B 模型能否学会格式。
2. 是否会过度敏感。
3. hard negative 是否有效。
4. 上下文反转样本是否被正确利用。
5. SFT 后是否值得进入 DPO/GRPO。

---

## 20. 标注禁忌

禁止：

1. 只因出现关键词就判 high。
2. 无上下文强行还原具体对象。
3. 把普通历史讨论判风险。
4. 把普通生活物品词判风险。
5. 把所有反讽都判风险。
6. 推理链只写支持证据，不写反证。
7. 使用“显然”“大家都懂”代替证据。
8. 让训练输出过长、过满、过度脑补。
9. 把同一模板改几个字后同时放进 train/test。
10. 把无法判断的样本硬标为 high。

## 21. 推荐工作流

```text
Step 1：按分类手册选定编码类别
Step 2：人工构造或采集样本
Step 3：填写基础字段
Step 4：写字面层、编码层、语境层分析
Step 5：写支持证据和反证
Step 6：标 risk_level
Step 7：标 ambiguity/evidence_strength/needs_context
Step 8：复核
Step 9：构造配对 hard negative
Step 10：进入训练集或评测集
```

## 24. 最终原则

本项目的数据标注要坚持一句话：

> 不训练模型“看见某词就害怕”，而训练模型“根据证据判断这句话是否真的在表达另一层意思”。

所有标注、训练、评测和奖励函数都应服务于这个原则。
